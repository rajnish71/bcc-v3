### Item 79a — Canonical Photo URL (self-correction + new-photo 404 fix)

**Governing document:** `PHOTO-ARCH-001` — canonical URL is `/showcase/photos/{numericId}`, never UUID.

#### The blunder (previous commit `90d73f1`)
The previous session changed `p.id` → `p.uuid` in `photographers/[username].astro` to build canonical links. This violated PHOTO-ARCH-001 and caused 404s because the static `[id].astro` route is built with numeric IDs only.

#### The actual original bug
Photos uploaded **after** the last build have no pre-built static page in `dist/`. The canonical photo page uses `getStaticPaths()` to build one page per photo at build time. A newly uploaded photo (e.g. id=762 "Long-tailed Shrike") had no static page, so navigating to `/showcase/photos/762/` returned 404.

#### Fix (commit `effbc67` + `d9e5973`)

**1. Reverted p.uuid → p.id** in `frontend/src/pages/photographers/[username].astro`:
- `openStory()` function: `photos.map(p => p.uuid)` → `p.id`
- `renderProfilePhotoCard()` click handler: `ph.uuid` / `p.uuid` → `ph.id` / `p.id`

**2. Canonical page reads ID from URL when serving as shell** (`frontend/src/pages/showcase/photos/[id].astro`):
```javascript
// Before (define:vars injected the ID, useless for a placeholder)
<script define:vars={{ photoId: id }}>
  const CANONICAL = `/showcase/photos/${photoId}`;

// After
<script define:vars={{ _builtId: id }}>
  const photoId = (_builtId && _builtId !== 'placeholder')
    ? _builtId
    : (window.location.pathname.replace(/\/$/, '').split('/').pop() || _builtId);
  const CANONICAL = `/showcase/photos/${photoId}`;
```
When Nginx serves the placeholder shell for a new photo, `_builtId` = `'placeholder'`, so the JS reads the actual numeric ID from `window.location.pathname`.

**3. Always build the placeholder page** (`getStaticPaths`):
```javascript
// Before: placeholder only added if no photos in DB (never triggered on prod)
if (!paths.length) paths.push({ params: { id: 'placeholder' } });

// After: always included so Nginx try_files has a valid file to serve
paths.push({ params: { id: 'placeholder' } });
```

**4. Added Nginx try_files block** for `/showcase/photos/` (same pattern as `/photographers/` from item 52):
```nginx
location /showcase/photos/ {
    root /var/www/bcc-v3/frontend/dist;
    try_files $uri $uri/ $uri/index.html /showcase/photos/placeholder/index.html;
}
```
Applied directly on server: `/etc/nginx/sites-available/bcc.bhopal.info`.  
Backup: `bcc.bhopal.info.bak.item79a`.

#### Verification (live browser)
- `/showcase/photos/762/` → "Long-tailed Shrike" by Rajnish Khare, full metadata ✓  
- `/showcase/photos/900/` (not in build, not in DB) → placeholder shell loads, reads `900` from URL, shows "Photograph not found or access restricted" gracefully ✓

### Item 78 — Aspect Ratio Violation in Gallery Grids

**Rule:** Never crop, never change aspect ratio. Original photographer composition is inviolable.

#### Root cause
`width_px` and `height_px` were NULL for **all** photos in the database (237 photos). The justified grid algorithm falls back to `aspect = 1.5` (landscape) when these are NULL. Portrait or panoramic photos got a landscape-shaped container, and `object-fit: cover` / default `fill` distorted or cropped them.

**Two symptoms visible in screenshot:**
- First photo: panorama being rendered in wrong container shape
- Second photo: vertical/portrait rendered as horizontal (landscape container)

#### Fix — 3 layers

**Layer 1: CSS safety net** — guarantees no distortion regardless of stored data

`frontend/src/styles/global.css` — `.photo-card__img`:
```css
/* Before */
object-fit: cover;   /* crops to fill container */

/* After */
object-fit: contain;  /* full image always visible, no crop, no stretch */
background: #FAF8F4;  /* page background fills any letterbox (invisible when ratio correct) */
```

`frontend/src/pages/index.astro` — `.jrow .photo-card img`:
```css
/* Added same safety net */
object-fit: contain;
background: #FAF8F4;
```

**Layer 2: Upload fix** — captures dimensions for all future uploads

`frontend/src/pages/hub/upload/index.astro`:
- Added `widthPx?: number; heightPx?: number;` to `ExifData` interface
- `extractExif()` now reads natural dimensions via browser `Image` API (works for JPEG, PNG, WebP, TIFF):
```javascript
const objUrl = URL.createObjectURL(file);
await new Promise<void>((resolve) => {
  const img = new Image();
  img.onload = () => { base.widthPx = img.naturalWidth; base.heightPx = img.naturalHeight; resolve(); };
  img.onerror = () => resolve();
  img.src = objUrl;
});
URL.revokeObjectURL(objUrl);
```
- Confirm payload now includes `width_px: item.exif?.widthPx, height_px: item.exif?.heightPx`

**Layer 3: Backfill script** — corrects existing photos

`scripts/backfill_photo_dimensions.js`:
- Fetches first 64 KB of each photo from ImageKit via HTTP range request
- Parses JPEG SOF markers or PNG IHDR to extract natural pixel dimensions
- Updates `photos.width_px` and `photos.height_px` in DB
- DRY_RUN=1 for preview

**Applied on prod:**
```bash
NODE_PATH=/var/www/bcc-v3/backend/node_modules \
DB_PASSWORD=$(grep DB_PASSWORD backend/.env | cut -d= -f2) \
node scripts/backfill_photo_dimensions.js
```
Result: **145 updated**, 1 skipped (unknown format), 91 failed (HTTP 400 — orphaned R2 uploads with no file behind them; these photos may need cleanup separately).

**Key data point:**  
- Photo 762 "Long-tailed Shrike": was `width_px=NULL, height_px=NULL` → now `1638×2048` (portrait, ratio 0.80). Previously shown in a 1.5-ratio landscape container, distorting it.
- Photo 761 "Thread of Dreams": `1365×2048` (portrait)

#### Verification
With correct dimensions stored, the justified grid sizes containers to match each photo's natural ratio. `object-fit: contain` then fills exactly with no letterboxing and no distortion. The contain fallback only becomes visible if a photo still has NULL dimensions (the 91 failed backfill cases).
