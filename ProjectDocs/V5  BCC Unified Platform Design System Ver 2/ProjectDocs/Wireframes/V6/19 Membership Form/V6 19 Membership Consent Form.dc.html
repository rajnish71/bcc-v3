Membership Application
& Renewal Form

Three-step form serving two distinct authenticated flows: Variant A (Apply) for registered users without active membership, Variant B (Renew) for members in the 60-day renewal window or with expired status. MEM-006 and MEM-007 constitutional constraints govern all copy throughout both variants.

Design reference only. Hub shell (header, sidebar, footer, responsive behaviour) inherited from V6 09–12. This document owns MembershipApplicationFlow and its states only.

01

Overview

Both variants share identical step structure and visual design. The form lives inside the HubLayout shell (authenticated, minimum USER role). MEM-006 constitutional constraints govern all public-facing copy throughout.

VARIANT A New Application

Membership Application

/hub/membership/apply · RBAC: USER · no active/pending membership
  • —All fields empty on load except Full Name (read-only) and City/State defaults
  • —Shown when authenticated user has no active or pending membership
  • —Creates a PENDING application record only (MEM-007)
  • —Success: "Application Submitted" — no membership number ever shown
VARIANT B Renewal

Membership Renewal

/hub/membership/renew · RBAC: MEMBER · renewal window or expired
  • —All fields pre-populated from existing member profile
  • —Shown when expires_at ≤ 60 days away or membership already expired
  • —Consent must be re-collected on every renewal — same T&C step
  • —Step 3 shows current expiry + projected new expiry after approval
Constitutional Constraints — Non-Negotiable
MEM-006 P1
Three CTAs always distinct: Sign In / Register / Apply for Membership. This is the Apply pathway only. Never conflate with registration or sign-in.
MEM-006 Public Domain
Only "Basic Membership" shown on this form. Full, Life, Patron, Founding Member class names must NOT appear here. Student/Individual visible only inside Hub after activation.
MEM-007
No permanent membership number on submission. BCCTempXXXXX assigned at admin activation only. Success screen never shows a membership number. State: PENDING only.

08

Constitutional Notes

These constraints are non-negotiable and enforced unconditionally at every layer of the platform. No future wireframe, product decision, or engineering change may override them without a formal constitutional amendment.

MEM-006 P1
Three CTAs always distinct
Sign In / Register / Apply for Membership are three separate, always-distinct calls to action. This form is the Apply pathway only. It must never be conflated with registration or with the Sign In flow.
FORBIDDEN: Any copy, button, or flow that implies membership is created on registration.
MEM-006 Public Domain
Membership class names hidden
Membership class names — Full, Life, Patron, Founding Member — must not appear anywhere on this public-facing form. Only "Basic Membership" is shown to applicants. Student and Individual class names are visible only inside the Hub after activation and are never exposed in the application flow.
FORBIDDEN: "Full Member", "Life Member", "Patron", "Founding Member" anywhere on this form.
MEM-007
No number on submission
A permanent membership number (BCC-XXXX) is never issued on form submission. The member receives BCCTempXXXXX only after an administrator manually activates the application. The success screen must show "Application under review" and never show any membership number — temporary or permanent.
FORBIDDEN: Any membership number (temporary or permanent) on the confirmation/success screen.
