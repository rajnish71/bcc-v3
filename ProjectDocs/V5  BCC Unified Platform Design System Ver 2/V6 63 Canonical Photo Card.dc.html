01

Component Purpose & Scope

The Photo Card is the atomic unit of photography presentation. It is the contract between a single photograph and the user. All gallery layouts, all photography contexts, and all roles consume exactly one card component — parameterised by variant and context, never by page or module.

PHOTO-ARCH-001 — Governing Constraint

One card. Everywhere.

There is no such thing as a "Showcase card", a "Portfolio card", or an "Admin card" at the design level. There is ONE Photo Card component, five variants. Variant selection is driven by context, not by page or feature. Any divergence from this architecture must pass design authority review.

Layout-agnostic.

The card does not know what layout contains it. It does not set its own width or height — those are supplied by the LayoutEngine. The card fills its container, respects the natural aspect ratio of the photograph, and applies its variant's visual treatment. Nothing more.

03

Visual Hierarchy

The photograph commands 100% of available attention at rest. Attribution and context emerge only on intent (hover). The hierarchy is not a typographic grid — it is a temporal sequence: image first, identity second, context third.

Priority 1 — Always
The Photograph
100% visual area at rest. No competing elements. No persistent UI chrome. The image is the card. Nothing fights for attention at the first moment of perception.
Priority 2 — On intent
Photographer Identity
Name in Outfit 11/600 white. Revealed only on hover. Largest text in the caption. Clicking the card resolves to this photographer's profile — identity is the primary action target.
Priority 3 — On intent
Context
Photo title and activity in JetBrains Mono 9px at 60% white. Tertiary — provides context without competing. JetBrains Mono's monospaced nature reads as metadata, not as prose. Deliberately understated.
Hierarchy Rules — Frozen
  • —No UI element may appear above the photograph at rest
  • —Caption bar always occupies bottom edge only
  • —Photographer name always larger than photo title
  • —Viewfinder corners never carry text
  • —Gold (#C9A961) used only in Featured variant eyebrow, never in Default/Compact
  • —No typography above mid-point of card in any variant
  • —Scrim overlay serves only readability — not identity
  • —Admin variant: hierarchy inversion allowed — status badge leads

Why fixed px not %
A corner bracket that grows with the card loses its calibration-instrument quality. At 16px it reads as a precision marker; at 40px on a large hero it reads as decoration.
Why 10px inset not 0px
Corner brackets at 0px read as a border treatment. At 10px they read as a framing device — a reference to the viewfinder, placed inside the image space.
Caption bar: natural height
Caption bar has no fixed height. Padding defines the spaciousness. Content (one or two lines) determines the actual height. Never force a min-height that clips text.

Image Presentation Rules

Non-negotiable. These rules are inherited from V6 00 Photography Philosophy and PHOTO-ARCH-001. They apply to every variant, every context, every layout. No exceptions without design authority amendment.

FORBIDDEN — In every context, every variant
  • ✕object-fit: cover on the photograph
  • ✕Fixed-height containers that force image crop
  • ✕border-radius on the photograph or card root
  • ✕scale() transform on hover (image or card)
  • ✕Watermark, text, or logo overlaid on photograph
  • ✕Decorative border or drop shadow on photo itself
  • ✕Colour-grading, filter, or tint on photograph
  • ✕Loading spinner over the photograph
REQUIRED — Every photograph
  • ✓Rendered at its original aspect ratio from EXIF metadata
  • ✓width: 100% · height: auto on the img element
  • ✓Served from ImageKit at original dimensions, no server crop
  • ✓alt text always present (see §13 Accessibility)
  • ✓Lazy loading on images below the fold
  • ✓Natural ratio skeleton during load (see loading state §10)
  • ✓display: block to eliminate baseline gap below img

Responsive Behaviour

The card itself is not responsive — its parent layout is. On mobile, hover captions become persistent labels because hover intent does not exist on touch. All other card behaviour is inherited from the layout engine's responsive mode.

Property Desktop ≥901px Tablet 481–900px Mobile ≤480px
{{ rr.prop }}
{{ rr.d }}
{{ rr.t }}
{{ rr.m }}
Touch / Mobile Caption Persistence
On touch devices (<768px, pointer: coarse), the caption is always visible — there is no hover state. The photographer name appears as a persistent bottom label. Activity subtitle is hidden to maintain visual breathing room at small sizes. The viewfinder corners are never shown on touch.
Card Sizing — Who Decides
The card component sets no width, no height, no min-height. The parent LayoutEngine provides the bounding box. The card fills it. If the LayoutEngine collapses to 1-col on mobile, the card becomes full-width — the image renders at its natural height for that width. The card never needs to know its own size.

Accessibility

WCAG 2.1 AA target. Photography presentation must not create barriers for users with visual impairments, motor limitations, or cognitive differences. The card's keyboard and screen reader experience must be equivalent to its visual experience.

Screen Reader — Required markup
<a href="/photos/[id]"
   role="article"
   aria-label="[Photo title] by [Photographer], [Activity]"
   tabIndex="0">
  <img
     src="[imagekit-url]"
     alt="[Photo title] by [Photographer name]"
     loading="lazy"
     width="[width]" height="[height]"/>
</a>
Alt text formula: "[Photo title] by [Photographer name]". If title is absent: "[Photographer name]'s photograph". Never leave alt empty on photos. Never use the filename or ID as alt text.
Focus & Keyboard
  • —:focus-visible shows 2px solid #C9A961 ring, outline-offset 2px
  • —Tab order follows DOM order (matches visual reading order)
  • —Enter / Space activates the card's primary action
  • —Caption visible on :focus (same as hover) for keyboard users
  • —Admin action buttons (✎, ☐) independently focusable: separate tabIndex
  • —Admin action buttons have aria-label: "Edit photo" / "Select photo"
Contrast check
White name on rgba(20,18,16,0.68) over dark photo: ≥ 4.5:1 (pass AA).
Gold #C9A961 on #141210: 7.2:1 (pass AAA).
White 60% on rgba overlay: minimum context only — screen reader must carry the information independently.

Non-token values — rationale required
Two values in the card are not standard tokens and are intentionally fixed:

Frozen Rules — Non-negotiable implementation constraints
  • 01The card never sets its own width or height. Parent LayoutEngine owns geometry.
  • 02img: width:100%; height:auto; display:block; border-radius:0. Always.
  • 03object-fit:cover is forbidden on the photograph element in every variant.
  • 04scale() transform is forbidden on hover in every variant.
  • 05All state transitions are CSS transitions only. No JS animation libraries on card states.
  • 06:focus-visible (not :focus) triggers the gold ring. Mouse clicks do not show focus ring.
  • 07alt text is mandatory. Photo title + photographer name. Never empty, never filename.
  • 08prefers-reduced-motion: disable all transitions. Keep state visibility. No animation.
  • 09Caption in Minimal variant is suppressed entirely — zero DOM elements, not hidden via CSS.
  • 10The card root element is the only interactive target (one link, one action). No nested links.
Astro component name
PhotoCard.astro
Single file. Variant prop selects rendering branch. Context prop switches between public and admin visual treatment. No per-page component files.
Data contract
PhotoCard accepts a single photo: Photo prop (as defined in V6 60 §08 data model). All metadata is derived from this object — no separate props for title, name, tags. Variant and context are separate control props.
What changes at V7
Phase 2 (V7 scope): Award badges · Like / save mechanics · Multi-select actions · Contest submission state. None of these amend the core card contract — they extend via new context prop values and new variant branches added by design authority amendment.
