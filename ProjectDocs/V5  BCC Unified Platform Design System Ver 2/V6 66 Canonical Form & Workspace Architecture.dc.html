Canonical Form &
Workspace Architecture

The UX handbook for every editing experience in the BCC Unified Platform. Defines six reusable workspace patterns. Reconciles with existing Member Hub UX and Design Constitution V6 99.

This document defines architecture, not implementation. It does not specify textboxes, dropdowns, checkboxes, or individual controls. Those belong to development.

01

Simple Form

Short, focused interaction. One task. Minimum friction. The user arrives knowing what they need to do.

Purpose

Complete a single, well-defined task in the fewest possible steps. No navigation, no sidebar, no distraction. The form is the entire page.

When to use
  • →Login & Sign Up
  • →Forgot Password / Reset Password
  • →OTP verification
  • →Newsletter subscribe
  • →Contact / Feedback
Guide Wireframe — Simple Form
  ┌────────────────────────────────────────────┐
  │  [ BCC Logo ]                              │  Page / Modal shell
  ├────────────────────────────────────────────┤
  │                                            │
  │    [ Form Title ]          ← Outfit H2     │
  │    [ Subtitle / instruction ]              │
  │                                            │
  │    ┌──────────────────────────────────┐    │
  │    │  Field 1                         │    │
  │    └──────────────────────────────────┘    │
  │    ┌──────────────────────────────────┐    │
  │    │  Field 2                         │    │
  │    └──────────────────────────────────┘    │
  │                                            │
  │    [ ●●●●●●●●●●●●  Primary Action ]        │  ← Gold CTA
  │                                            │
  │    [ Secondary text link ]                 │
  │                                            │
  ├────────────────────────────────────────────┤
  │  Max width: 480px · Centred · No sidebar   │
  └────────────────────────────────────────────┘
User Flow
  1. Arrive at isolated page or modal
  2. Fill fields (1–5 fields maximum)
  3. Submit — single gold CTA
  4. Inline success or redirect
Composition Rules
  • —Max 480px form width
  • —Centred on page or in modal
  • —No navigation or sidebar
  • —One gold CTA — full width
  • —No autosave (submit only)
Responsive
  • —Unchanged on mobile
  • —16px padding sides ≤480px
  • —Form fills viewport width
  • —Keyboard lifts no reflow
02

Sectioned Form

Large settings pages with many fields, organised into named groups. The user reads the whole page and edits where needed — not a sequential flow.

Purpose

Expose a large configuration surface grouped into logical sections. The user can scan, jump to any section, and save independently or all at once.

When to use
  • →Member Account Settings
  • →Event Settings
  • →Project Settings
  • →Admin configuration panels
Guide Wireframe — Sectioned Form
  ┌──────────────────────────────────────────────────────────────────┐
  │  [ Page Title ]  [ Subtitle ]                    [ Save All ]    │  ← Page header
  ├────────────┬─────────────────────────────────────────────────────┤
  │            │                                                      │
  │  SECTION   │  ┌──────────────────────────────────────────────┐   │
  │  ANCHORS   │  │  ── Section A ──────────────────────────────  │   │
  │            │  │     [ Field ]  [ Field ]                      │   │
  │  A         │  │     [ Field ]                                 │   │
  │  B         │  │                         [ Save Section A ]    │   │
  │  C         │  └──────────────────────────────────────────────┘   │
  │  D         │                                                      │
  │            │  ┌──────────────────────────────────────────────┐   │
  │  (sticky)  │  │  ── Section B ──────────────────────────────  │   │
  │            │  │     [ Field ]  [ Field ]                      │   │
  │            │  │                         [ Save Section B ]    │   │
  │            │  └──────────────────────────────────────────────┘   │
  │            │                                                      │
  │  200px     │  Max content width: 720px · Right of anchor rail    │
  └────────────┴─────────────────────────────────────────────────────┘

  Mobile: anchor rail collapses to sticky top tab strip
          sections stack full-width beneath
User Flow
  1. Arrive at settings page
  2. Scan section anchors
  3. Jump to relevant section
  4. Edit fields in place
  5. Save section (or all)
Composition Rules
  • —Left sticky anchor rail (200px)
  • —Content max-width 720px
  • —Each section saves independently
  • —Page-level Save All optional
  • —Danger zone section at bottom
Responsive
  • —≤900px: rail → sticky tab strip
  • —≤640px: single column full-width
  • —Sections stack vertically
  • —Save anchored to bottom sheet
Platform Example — V6 20 Account Settings

The existing V6 20 Account Settings page is the canonical instance of this pattern. Sections: Profile, Contact, Password, Notifications, Privacy, Membership, Danger Zone. The anchor rail and per-section save model defined there is the reference implementation.

03

Profile Editing Workspace

The primary editing experience of the platform. The workspace where a member curates their identity — biography, photography, public presence. Rich and persistent.

Purpose

The member edits their public identity — biography, avatar, cover, social links, interests, member statement. Changes are visible on their public profile immediately on save. The workspace reconciles with the existing Member Hub.

When to use
  • →Member Profile edit
  • →Mentor Profile edit
  • →Volunteer Profile edit
  • →Admin Profile edit

  │  │  [ Display Name ]  [ Member Since ]  [ Role badge ]        │  │
  │  └────────────────────────────────────────────────────────────┘  │
  │                                                                   │
  │  ┌─────────────────────────────┐  ┌────────────────────────────┐ │
  │  │  IDENTITY                   │  │  PREVIEW                   │ │
  │  │  ─────────────────────────  │  │                            │ │

  │  └─────────────────────────────┘                                  │
  │                                                                   │
  │  Left col max 640px · Right preview col ~320px                    │

User Flow
  1. Enter via Hub → Profile tab
  2. Cover / avatar replace inline
  3. Edit identity fields
  4. Preview updates live
  5. Save Profile — single action
Composition Rules
  • —Lives inside Hub shell
  • —Cover & avatar edit inline
  • —Live preview column (optional)
  • —Grouped: Identity + Photography
  • —Autosave draft, explicit save
Responsive
  • —Preview col hidden ≤900px
  • —Edit col full-width ≤640px
  • —Save floats bottom on mobile
  • —Hub rail → top tabs ≤1199px
Reconciliation — V6 13 Member Hub Profile

This pattern reconciles with the existing V6 13 Members Hub Profile page. The Hub shell and tab navigation remain unchanged. The Profile Editing Workspace occupies the main content area under the Profile tab only. Do not redesign the Hub shell to accommodate this pattern.

04

Media Upload Workspace

The workflow for bringing photographs and documents into the platform. Focus is on the authoring experience — titling, captioning, categorising — not on the upload widget itself.

Purpose

Upload one or many media files, add metadata (title, caption, category, EXIF override), then publish or save as draft. The workspace honours the natural aspect ratio of every photograph throughout.

When to use
  • →Photo upload to portfolio
  • →Gallery / album upload
  • →Contest submission
  • →Avatar & cover image replace
  • →Document & consent form upload
Guide Wireframe — Media Upload Workspace (multi-photo)
  ┌──────────────────────────────────────────────────────────────────┐
  │  [ Upload Photos ]         [ Draft ]  [ Publish ]                │  ← Action bar
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                  DROP ZONE                                   │ │
  │  │        Drag photos here or [ Browse ]                       │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                                                                   │
  │  QUEUE ── 3 photos ──────────────────────────────────────────    │
  │                                                                   │
  │  ┌──────────────┐  ← selected    ┌──────────┐  ┌──────────┐     │
  │  │              │                │          │  │          │     │
  │  │   [image]    │  EDIT PANEL →  │  [img2]  │  │  [img3]  │     │
  │  │              │  Title         │          │  │          │     │
  │  │  ▪ Selected  │  Caption       │  ◦       │  │  ◦       │     │
  │  └──────────────┘  Category      └──────────┘  └──────────┘     │
  │                    Tags                                           │
  │                    EXIF (read-only extract)                       │
  │                    Visibility                                     │
  │                                                                   │
  │                    [ Apply to all ] · [ Save this photo ]         │
  │                                                                   │
  │  Queue thumbnails at natural ratio · Edit panel right of queue   │
  └──────────────────────────────────────────────────────────────────┘
Guide Wireframe — Single Asset Upload (avatar / cover)
  ┌──────────────────────────────────────┐
  │  [ Replace Avatar ]                  │
  │  ─────────────────────────────────── │
  │                                      │
  │    ┌──────────────────────────────┐  │
  │    │   Current image preview      │  │
  │    └──────────────────────────────┘  │
  │                                      │
  │    [ Upload new ]  [ Remove ]        │
  │                                      │

  Delivered as a modal (Quick Edit pattern).
  No full-page shell for single-asset replacement.
User Flow (multi)
  1. Drop or browse files
  2. Queue builds with previews
  3. Select photo → edit panel
  4. Title, caption, category
  5. Repeat or Apply to All
  6. Publish / Save draft
Composition Rules
  • —Thumbnails at natural ratio
  • —Edit panel right of queue
  • —Progress per file, not global
  • —Unsaved photos clearly flagged
  • —Draft auto-persists on navigate
Responsive
  • —≤768px: queue above edit panel
  • —Queue scrolls horizontally
  • —Edit panel full-width below
  • —Publish bar fixed bottom
Platform Examples — V6 11 Members Hub Upload

The existing V6 11 Members Hub Upload page is the canonical instance of this pattern. All multi-photo upload flows across the platform should reconcile with that design before introducing new variations.

05

Wizard Workspace

Long multi-step processes that should not be presented all at once. The wizard breaks complexity into committed, sequential stages with clear progress.

Purpose

Guide the user through a process that requires sequential decision-making — where a later step depends on an earlier one, or where seeing all fields at once would be overwhelming and discouraging.

When to use
  • →Membership application
  • →Contest entry
  • →Event registration
  • →Volunteer / Mentor application
  • →Workshop or course enrolment
Guide Wireframe — Wizard Workspace
  ┌──────────────────────────────────────────────────────────────────┐
  │  [ BCC Logo ]                                      [ Save & Exit]│
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   Step 1 ──●── Step 2 ──○── Step 3 ──○── Step 4 ──○── Review    │  ← Progress rail
  │   Complete    Active       Pending      Pending      Pending      │
  │                                                                   │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [ Step Title ]               ← Outfit H2                       │
  │   [ Step instruction ]         ← Inter 16px                      │
  │                                                                   │
  │   ┌──────────────────────────────────────────────────────────┐   │
  │   │  Step content                                             │   │
  │   │                                                           │   │
  │   │  Fields / choices / uploads relevant to this step        │   │
  │   │                                                           │   │
  │   └──────────────────────────────────────────────────────────┘   │
  │                                                                   │
  ├──────────────────────────────────────────────────────────────────┤
  │  [ ← Back ]                           [ Continue → ]             │  ← Step footer
  └──────────────────────────────────────────────────────────────────┘

  Final step is always Review — shows all answers before submit.
  Max content width: 680px · Centred
User Flow
  1. Land on Step 1
  2. Complete each step in order
  3. Back always available
  4. Save & Exit saves draft
  5. Review step before submit
  6. Confirmation state
Composition Rules
  • —Isolated shell (no SiteHeader)
  • —Progress rail always visible
  • —One primary action per step
  • —Review step before final submit
  • —Draft auto-saves on each step
  • —Max 6 steps before splitting
Responsive
  • —Progress rail scrolls horizontal
  • —Step labels hide, dots remain
  • —Content full-width ≤640px
  • —Footer nav fixed bottom
Membership Application Steps
  1. Personal Details
  2. Photography Background
  3. Sample Photos (3–5)
  4. Consent & Rules Agreement
  5. Review & Submit
Contest Entry Steps
  1. Select Category
  2. Upload Photo(s)
  3. Title, Caption, Statement
  4. Review & Submit
06

Quick Edit Workspace

Small, immediate edits that do not warrant a full page. Delivered as a modal or sheet overlay. Focused, fast, dismissable.

Purpose

Handle a discrete, low-stakes action without navigating away from the current context. The user triggers it, acts, and returns. Confirmation dialogs, rename, and status changes all live here.

When to use
  • →Rename photo / album / project
  • →Delete with confirmation
  • →Change visibility / status
  • →Avatar / cover quick replace
  • →Share / copy link
  • →Archive or unpublish
Quick Edit Modal
  ┌────────────────────────────────┐
  │  [ Title ]             [ × ]  │
  ├────────────────────────────────┤
  │                                │
  │  [ Context — what you're       │
  │    editing ]                   │
  │                                │
  │  ┌──────────────────────────┐  │
  │  │  Field / Choice          │  │
  │  └──────────────────────────┘  │
  │                                │
  │  [ Cancel ]  [ Confirm ]       │
  │                                │
  └────────────────────────────────┘

  Max width: 480px
  Overlay backdrop: ink-900 @ 40%
Destructive Confirmation
  ┌────────────────────────────────┐
  │  Delete Photo?        [ × ]   │
  ├────────────────────────────────┤
  │                                │
  │  This cannot be undone.        │
  │  "Monsoon at the Fort"         │
  │  will be permanently removed.  │
  │                                │
  │  Type CONFIRM to proceed:      │
  │  ┌──────────────────────────┐  │
  │  │                          │  │
  │  └──────────────────────────┘  │
  │                                │
  │  [ Cancel ]  [ Delete ⚠ ]      │
  └────────────────────────────────┘

  Destructive action: danger colour.
  Requires typed confirmation.
User Flow
  1. Trigger from context menu / button
  2. Modal opens, context preserved
  3. Single action (0–1 fields)
  4. Confirm or Cancel
  5. Modal closes, page updates
Composition Rules
  • —Always a modal, never a page
  • —Max 480px wide
  • —Cancel always present
  • —Esc closes without saving
  • —Destructive: typed CONFIRM
  • —No autosave in quick edits
Responsive
  • —Modal → bottom sheet ≤640px
  • —Sheet slides up from bottom
  • —Swipe down to dismiss
  • —Full-width buttons on mobile
07

Shared Principles

These principles apply across all six workspace patterns. They are not repeatable guidelines — they are mandatory behaviours that every editing experience in the platform must honour.

Autosave
Autosave & Unsaved Changes
Autosave applies to
  • ✓Profile Editing Workspace
  • ✓Sectioned Form (per-section)
  • ✓Wizard (on step completion)
  • ✓Media Upload (draft queue)
Autosave does not apply to
  • ×Simple Form (submit only)
  • ×Quick Edit (confirm or cancel)
  • ×Destructive confirmations
Unsaved Changes Guard

When a user attempts to navigate away from a workspace with unsaved changes, a Quick Edit confirmation modal intercepts: "You have unsaved changes. Leave without saving?" with Cancel and Leave options. This applies to all patterns where autosave is not active.

Validation
Validation Philosophy
1
Validate on blur, not on keystroke
Show errors after the user has left the field. Never interrupt mid-entry. Exception: real-time feedback for password strength is permitted.
2
Error messages are instructive, not punitive
Tell the user what to do, not what they did wrong. "Enter a valid email address" not "Invalid email."
3
Server errors surface in context
API errors appear beside the relevant field — not only in a top-of-page banner. Banners are for errors that do not map to a specific field.
4
Never block progress on optional fields
Required fields are clearly marked. Optional fields are never the reason a form cannot be submitted. Wizards never block a step for optional fields.
Permissions
Permissions & Read-Only Mode

Every workspace has an implicit read-only state. When a user does not have permission to edit the content, the workspace renders identically — but all fields are non-interactive and the save actions are absent or disabled.

  Read-only indicators:
  — No edit affordances (no cursor change on field hover)
  — Input fields render as display text
  — Action buttons absent (not disabled — absent)
  — A small label: "VIEW ONLY — You do not have 
    permission to edit this content."

Admins viewing another member's profile see read-only mode with an additional Edit as Admin action that elevates permission contextually — not a separate page.

Progress
Progress & Loading States
Saving
Button enters loading state. Label changes to "Saving…". Spinner replaces icon. Never disable the whole form.
Saved
Brief confirmation: checkmark + "Saved" for 2 seconds. Then returns to normal state. No toast for autosave — only for explicit save.
Upload Progress
Per-file progress bar beneath thumbnail. Overall count shown in action bar. Never a global progress that hides file-level detail.
Wizard Progress
The step rail is the canonical progress indicator. No separate progress bar. Completed steps show a checkmark. Current step highlighted.
Accessibility
Accessibility & Keyboard Navigation

All workspace patterns must be fully operable by keyboard. Focus order follows the visual reading order. Focus states use the canonical 2px gold outline (V6 99 §15).

Key Behaviour
TabAdvance through interactive elements
Shift+TabReverse through interactive elements
Enter / SpaceActivate focused button or control
EscClose modal / dismiss Quick Edit without saving
Arrow keysNavigate within radio groups, wizard steps, upload queue

Modal and Quick Edit workspaces must trap focus within the modal while open. Focus returns to the triggering element on close. ARIA roles: dialog, aria-labelledby pointing to the modal title.

Mobile
Mobile Adaptation
Simple Form
Unchanged. Fills viewport width. 16px horizontal padding.
Sectioned Form
Anchor rail collapses to horizontal scroll strip at top. Sections stack single column. Save anchored to bottom.
Profile Workspace
Preview column hidden. Edit column full width. Save button fixed at bottom.
Media Upload
Queue scrolls horizontal above edit panel. Edit panel full width below. Publish bar fixed bottom.
Wizard
Progress rail shows dots only (no labels). Step nav fixed at bottom. Single column content.
Quick Edit
Modal becomes bottom sheet. Slides up from bottom edge. Swipe down to dismiss. Full-width action buttons.

Pattern Selection Guide

When designing a new editing experience, use these criteria to select the correct workspace pattern. Do not invent new patterns — adapt an existing one.

Decision Tree
  New editing experience required
  │
  ├─ Is it a single, isolated task with ≤5 fields?
  │   └─ YES →  01 Simple Form
  │
  ├─ Is it a small, immediate action without leaving context?
  │   └─ YES →  06 Quick Edit (modal)
  │
  ├─ Is it a large page of settings organised into sections?
  │   └─ YES →  02 Sectioned Form
  │
  ├─ Is it a member editing their public identity / presence?
  │   └─ YES →  03 Profile Editing Workspace
  │
  ├─ Is it bringing media files into the platform?
  │   └─ YES →  04 Media Upload Workspace
  │
  ├─ Is it a multi-step sequential process (membership, contest)?
  │   └─ YES →  05 Wizard Workspace
  │
  └─ None match → Consult design authority before creating a new pattern
