Unified Upload Studio

The photographer's ingestion workspace. Inherits the Hub shell from V6 09 exactly — SiteHeader, 240px left rail, fluid main content, SiteFooter — and the Portfolio Wall's visual vocabulary from V6 10. This is not an admin media library: photography leads, metadata is contextual, administration is invisible until asked for.

One workflow, any quantity. A single photograph and a batch of two hundred pass through the identical Drop Zone → Queue → Metadata → Publication path. There is no separate "single upload" mode — the workflow simply has one item in it.

Design reference only — no implementation, no tokens, no CSS. The Hub shell is not redesigned here; this pass strengthens annotations, lifecycle, component tree, and specification only. Navigation (rail, tabs, mobile bar) is not owned by this document — it is inherited from V6 12 Members Hub Navigation via HubLayout / HubSidebar. This pass also removes the duplicated rail markup previously sketched inline.

Every tile is a photograph, never a table row. Lifecycle: Queued (55% opacity, no chip motion) → Uploading (gold progress ring over the dimmed thumbnail, % chip) → Processing (soft blur, quiet neutral chip) → Complete (✓ chip, full clarity) → Failed (isolated — rust outline, plain-language reason, per-tile Retry) → Duplicate Detected (tan outline, matched file reference, Replace / Keep Both / Skip). Duplicate detection, like EXIF extraction, is automatic and never blocks the rest of the batch from uploading. "Cancel remaining" is the only batch action; it never touches completed files.

Once uploads complete, photographs move here for captioning. Metadata lives in a contextual Inspector — never a dominant form. Selecting several photographs enables bulk fields, applied only to the selected set — the rest of the batch is untouched. Any field where selected photos already differ shows "Mixed" and asks before overwriting. Visibility, Copyright, and Licensing arrive pre-filled from the member's profile defaults; the photographer may override any of them per photograph or per selection.

Bulk fields that already differ across the selection render as italic "Mixed" placeholders — typing into one requires an explicit click to confirm overwrite; nothing is silently replaced, and edits apply only to the selected photographs, never the whole upload. EXIF is always shown per-photograph (from the anchor image), auto-extracted on upload, never bulk-edited, and never required. 07

Specification

SECTION ANNOTATIONS
① Drop Zone — sole element of the empty state. Dashed gold frame, single icon, single CTA, quiet format hints. One photograph or many — same zone, same flow.
② Upload Queue — grid of photograph tiles, each carrying its own lifecycle, including automatic EXIF extraction and duplicate detection. No table, no rows.
③ SelectionBar — identical vocabulary to V6 10's Portfolio SelectionBar, repurposed for bulk metadata entry instead of batch ops. Applies only to the selected subset.
④ Inspector Panel — right-edge overlay on desktop, bottom sheet on tablet/mobile. Bulk fields show "Mixed" rather than silently overwriting. Carries Origin Activity, auto-extracted EXIF, profile-defaulted Visibility/Copyright/Licensing.
⑤ Publication — the UploadInspector's terminal action: Publish Now or Save as Draft, not a generic "Apply."
⑥ Complete State — plain sentence + split count (published / draft), two actions, one gold. No animation, no graphic.
⑦ QueueToolbar — appears only during active upload. Shows batch count, background-upload hint, and the single non-destructive batch action: Cancel Remaining. Never touches completed or failed tiles.
UPLOAD LIFECYCLE
Queued → visible, 55% opacity, no chip motion
Uploading → gold progress ring, % chip
Processing → soft blur, neutral chip
EXIF Extraction → automatic, silent, non-blocking
Duplicate Check → Already Uploaded → Replace / Keep Both / Skip
Complete → full clarity, ✓ chip
Failed → isolated, rust outline, plain-language reason, per-tile Retry
Uploads continue in the background on navigation. A failure, missing EXIF, or a detected duplicate never blocks or pauses sibling uploads.
DEFAULTS & INHERITANCE
Visibility ← Member Profile
Copyright ← Member Profile
Licensing ← Member Profile
Origin Activity ← manual, optional
All three profile defaults are overridable per photograph or per selection. Origin Activity, once set, cascades to every photograph in the upload — it feeds Showcase filtering, event galleries, photographer activity, and statistics.

SiteFooter
? = conditional
DropZone renders only when the queue is empty. QueueToolbar renders during active upload. UploadQueue renders once files have been added. SelectionBar renders only when photographs are selected in the metadata phase. UploadInspector renders only when photographs are selected and all selected items have finished uploading. SourcePickerSheet renders on ≤640px when the member taps "Choose Photographs". CompleteState replaces the entire workspace once all queued items have resolved.
Component ownership
QueueTile owns its own lifecycle independently — uploading, processing, complete, failed, duplicate. One tile's failure never pauses or blocks its siblings.
Automatic, non-blocking
EXIF extraction and duplicate detection run per-file, immediately on upload. Neither can interrupt or fail the upload itself. Missing EXIF and unresolved duplicates are flagged for later attention, never before upload completes.
UploadInspector vs. Portfolio InspectorPanel
These are distinct components sharing a visual shell. UploadInspector handles bulk metadata entry across a selection — not single-photo editing. Mixed-value fields show "Mixed" rather than silently overwriting. Publication is its terminal action. Portfolio InspectorPanel (V6 10) handles single-photo editing, navigation, and focal-point setting.
SelectionBar variants
Upload SelectionBar: count + clear only — no batch-ops actions.
Portfolio SelectionBar (V6 10): Feature · Visibility · Move · Delete.
Visually identical shell; behaviourally distinct.
