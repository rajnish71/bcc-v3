Six components. Each page assembles only what it needs. The system is additive — start with SearchBar alone, add QuickFilterChips for immediate access to the most-used filters, then layer FilterBar, SortControl, and Active Filters as the module requires.

Purpose & Usage
→Surface the 4–8 most-used filter values for the module. Never more than 8 chips — use FilterBar dropdowns for the remainder.
→Single-select by default. Opt into multi-select when the use case supports AND or OR combinations.
→Filtering is instant — results update as a chip is activated. No submit button required.
→Operates independently from FilterBar. A chip selection may co-exist with an active dropdown filter.
Responsive & Accessibility
→Mobile. Chip row scrolls horizontally below 768px. A fade-out gradient on the right signals overflow. No line wrapping.
→Desktop. Chips wrap to a second line only when the chip set exceeds one row. Maximum two lines.
→Keyboard. Each chip is a focusable button. Space or Enter toggles. Tab moves between chips in order.
→ARIA. Wrap row in role="group" aria-label. Each chip uses aria-pressed to communicate toggle state.
Spacing & Visual Hierarchy
Vertical position
Sits 8px below SearchBar, 16px above FilterBar. Never on the same row as either component.
Chip spacing
6px gap between chips. 14px horizontal padding per chip. 32px chip height.
Visual weight
Lighter than FilterBar dropdowns. 32px height vs FilterBar's 36px establishes clear visual hierarchy.

Implementation Notes

Behavioural and technical constraints for all implementations of this system.

Behaviour Rules
1
No page reload. Search and filter updates are applied client-side. Results update without navigation.
2
Filters combine. All active filters apply simultaneously as AND conditions.
3
Sort is independent. Changing sort order never clears active filters or resets search.
4
Clear All restores default. Returns search, filters, and sort to their initial page state.
5
Debounced input. SearchBar fires after 300ms pause — not on every keystroke.
